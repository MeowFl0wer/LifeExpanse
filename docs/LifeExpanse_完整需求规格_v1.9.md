## 1. 项目定位

本项目是一个融合以下能力的综合性 Web 平台：

- 个人知识管理（PKM）
- 日记、片语、笔记与文章创作
- 人生轨迹、地点足迹和飞行轨迹记录
- 公开个人主页与内容发布
- 支持多个密码入口的独立加密互动空间
- 图片、小视频、外部链接以及完整的数据导出、备份和恢复
- 注册、登录、用户隔离与后续公众开放能力

平台应当同时服务两种场景：

1. **个人私密记录**：用户记录自己的生活、日记、知识、地点和飞行数据，默认仅自己可见。
2. **公开内容展示**：用户可将部分内容公开，形成个人主页、博客和公开知识库。

## 2. 产品阶段

### 2.1 第一阶段：个人稳定运行

- 系统部署在可联网访问的服务器上，而不是只运行在某一台个人设备中。
- 只有站点主人账号。
- 可通过电脑、手机和平板等任意设备打开网页、登录并记录内容。
- 所有数据统一保存在服务器端，设备本地只保留必要的登录会话、缓存和未提交草稿。
- 支持全部记录、地图、飞行轨迹、独立加密空间和备份恢复。
- 可以不开放注册，或将注册模式设为“关闭”。
- 优先保证服务器数据安全、跨设备同步、可恢复和移动端可用。

### 2.2 第二阶段：邀请制开放

- 开放邀请码注册。
- 每个注册用户拥有独立数据空间和公开个人主页。
- 增加用户配额、后台管理、举报和风控能力。
- 验证多用户隔离、性能和备份策略。

### 2.3 第三阶段：公众开放

- 开放邮箱注册。
- 启用邮箱验证、密码找回、验证码、限流、内容举报和用户封禁。
- 支持平台级公开内容发现，但不影响用户私密数据隔离。
- 根据访问量升级数据库、对象存储、缓存和异步任务系统。

## 3. 用户角色与权限

### 3.1 超级管理员

- 管理站点配置、注册模式、用户、邀请码和系统备份。
- 查看平台运行状态和安全日志。
- 冻结、禁用或删除违规账号。
- 隐藏或删除公开违规内容。
- 不应默认直接查看用户私密内容；如确需维护访问，应有明确审计记录。

### 3.2 注册用户

- 创建和管理自己的随想、日记、笔记与文章、人生轨迹、足迹和飞行记录。
- 控制每条内容的可见性。
- 拥有个人公开主页。
- 导出、恢复或删除自己的数据。
- 管理自己的独立加密互动空间和访问密码。

### 3.3 普通访客

- 无需登录即可浏览公开内容。
- 不可访问任何私密内容。
- 输入某个用户设置的专属密码后，可临时访问该用户的独立加密互动空间，并按权限回复。

### 3.4 加密空间访客

- 不属于正式注册用户。
- 通过专属密码换取短期访问会话。
- 只能查看对应加密空间。
- 可以回复，但不能发布主内容。
- 回复昵称不是必填项；访客可以匿名回复，也可以主动填写昵称。

## 4. 注册与登录系统

### 4.1 注册模式

后台可配置三种模式：

1. **关闭注册**：仅管理员手动创建账号。
2. **邀请码注册**：输入有效邀请码后才可注册。
3. **公开注册**：任何人可注册，但必须通过邮箱验证和反滥用校验。

### 4.2 注册字段

最低字段：

- 用户名
- 邮箱
- 密码
- 确认密码
- 同意用户协议与隐私政策

可选字段：

- 显示名称
- 头像
- 个人简介
- 时区
- 默认语言

### 4.3 登录方式

首期支持：

- 邮箱或用户名 + 密码
- 保持登录
- 退出登录
- 忘记密码
- 邮箱重置密码

后续可增加：

- Passkey
- GitHub / Google 等 OAuth 登录
- 双因素认证（2FA）

### 4.4 安全要求

- 密码只保存强哈希，推荐 Argon2id。
- 登录会话优先使用 `HttpOnly + Secure + SameSite` Cookie。
- 不将长期有效登录令牌直接存入 localStorage。
- 登录、注册、密码重置和回复接口必须限流。
- 连续失败登录后触发临时锁定或验证码。
- 支持查看和退出其他设备会话。
- 关键操作记录审计日志。
- 用户删除账号前应二次确认，并提供数据导出入口。

## 5. 内容可见性模型

随想、日记、笔记与文章、人生轨迹等普通内容统一支持以下状态：

| 状态 | 含义 |
|---|---|
| 私密 | 仅内容所有者登录后可见 |
| 公开 | 无需登录即可访问 |
| 草稿 | 仅作者可见，不进入正式时间线或公开页面 |

普通内容不提供“单条内容密码保护”功能。密码访问只用于第 15 章定义的独立加密互动空间，并且密码作用于整个空间。

建议每条内容至少包含：

- 所属用户 `user_id`
- 内容类型
- 标题
- Markdown 原文
- 渲染摘要
- 可见性
- 发布时间
- 创建时间
- 修改时间
- 标签
- 封面图或附件
- 归档状态
- 软删除状态

所有公开查询必须在后端强制过滤可见性，不能只依赖前端隐藏。

## 6. 公开主页与游客端

### 6.1 用户公开主页

平台只使用一个主域名，例如：

`https://life.555978.xyz`

采用根路径用户名路由：

- `/{username}`：用户个人主页
- `/{username}/pkm`：笔记与文章统一入口
- `/{username}/thoughts`：公开随想（原创与摘录）
- `/{username}/diary`：用户选择公开的日记
- `/{username}/trajectory`：用户选择公开的人生轨迹
- `/{username}/map`：公开足迹地图
- `/{username}/flights`：公开飞行记录与航线图
- `/{username}/space`：该用户的多密码加密空间统一入口

示例：

- `https://life.555978.xyz/euan`
- `https://life.555978.xyz/euan/pkm`
- `https://life.555978.xyz/euan/thoughts`
- `https://life.555978.xyz/euan/space`

URL 路径建议统一使用小写。用户可以显示为 `Euan`，但规范用户名保存为 `euan`。访问 `/Euan/PKM` 时，系统应自动重定向到规范地址 `/euan/pkm`，避免大小写产生重复页面。

### 6.2 页面结构

顶部导航建议包含：

- 首页
- 笔记与文章
- 片语
- 生活轨迹
- 足迹地图
- 飞行轨迹
- 关于

首页可以混合展示所有公开内容，也可以按内容类型筛选。

### 6.3 公开内容要求

- 支持 SEO 基础信息。
- 支持分享链接和 Open Graph 预览。
- 支持按时间、标签和分类筛选。
- 支持移动端阅读。
- 公开 Markdown 必须进行 XSS 安全清洗。
- 用户可关闭自己的公开主页，只保留私密使用。

- `/{username}` 及其所有子页面只展示该用户名下的内容。
- 用户主页中不得插入其他用户的推荐文章、热门内容或跨用户搜索结果。
- 当前版本不提供全平台公开内容聚合页，也不提供跨用户内容搜索。
- 用户名查找可以作为独立功能存在，但查找到用户后只进入该用户自己的主页。


### 6.4 根域名与站点主人主页

直接访问：

`https://life.555978.xyz`

应展示站点主人（例如 `euan`）的公开主页，而不是平台用户目录或登录页。

同时保留站点主人的标准用户地址：

`https://life.555978.xyz/euan`

推荐实现方式：

- `/` 作为站点主人公开主页的首页别名。
- `/euan` 作为统一用户路由下的标准地址。
- 两个地址展示同一份公开数据。
- HTML 中设置统一 canonical 地址，避免搜索引擎把它们识别为重复页面。
- 站点主人可在后台修改根路径默认展示的账号，但系统必须确保同一时间只能配置一个根主页账号。

### 6.5 路由命名与保留字

采用 `/{username}` 路由时，必须维护系统保留路径，禁止用户注册为这些用户名。

至少保留：

- `login`
- `logout`
- `register`
- `signup`
- `app`
- `dashboard`
- `admin`
- `api`
- `settings`
- `account`
- `assets`
- `static`
- `uploads`
- `health`
- `search`
- `about`
- `terms`
- `privacy`

用户名建议：

- 全局唯一，忽略大小写。
- 长度 3–30 个字符。
- 首期仅允许小写英文字母、数字、短横线和下划线。
- 不能以短横线或下划线开头或结尾。
- 显示名称与用户名分开保存，显示名称可使用中文和大小写。
- 修改用户名后，旧地址至少保留一段时间并重定向到新地址。
- 已删除或停用的用户名应进入保留期，不能立即被其他人注册。

### 6.6 公共页面与私有工作区

`/{username}/...` 主要表示某个用户的内容空间。

同一页面可以根据访问身份展示不同内容：

- 未登录访客：只看到公开内容。
- 空间主人登录：可以看到自己的公开和私密内容，并显示编辑、新建和管理入口。
- 其他已登录用户：仍只能看到目标用户公开给他们的内容。

但平台级账号管理、注册登录和系统管理不应放在用户名路径下，建议使用：

- `/login`
- `/register`
- `/account`
- `/app`
- `/admin`

其中 `/app` 可以作为当前登录用户的统一私人工作台，再跳转或映射到其用户名模块页面。

### 6.7 推荐完整路由表

| 路由 | 用途 | 访问规则 |
|---|---|---|
| `/` | 站点主人公开主页 | 公开 |
| `/{username}` | 用户公开主页 | 公开或用户关闭 |
| `/{username}/pkm` | 用户知识管理入口 | 游客看公开内容，主人看全部 |
| `/{username}/thoughts` | 随想（原创与摘录） | 按内容权限 |
| `/{username}/diary` | 日记 | 按内容权限 |
| `/{username}/pkm` | 笔记与文章 | 按内容权限 |
| `/{username}/trajectory` | 人生轨迹 | 按内容权限 |
| `/{username}/map` | 足迹地图 | 按用户公开设置 |
| `/{username}/flights` | 飞行记录与轨迹 | 按用户公开设置 |
| `/{username}/space` | 多密码加密空间入口 | 密码或主人登录 |
| `/app` | 当前用户私人工作台 | 登录 |
| `/login` | 登录 | 公开 |
| `/register` | 注册 | 根据注册模式 |
| `/account` | 账号安全和资料 | 登录 |
| `/admin` | 平台管理后台 | 超级管理员 |
| `/api/v1/...` | 后端 API | 按接口鉴权 |

## 7. 登录后的统一工作台

### 7.1 顶部导航

登录后顶部导航常驻，包含：

- 总览
- 片语
- 日记
- 笔记与文章
- 人生轨迹
- 足迹地图
- 飞行日志

- 搜索

### 7.2 全局创作入口

全站提供统一“新建”按钮，可使用顶部按钮或右下角悬浮按钮。

点击后展示：

- 记随想
- 写日记
- 新建笔记
- 写文章
- 记录人生轨迹
- 发布独立空间内容

支持键盘快捷键快速唤起。

### 7.3 总览页

展示：

- 今日是否写日记
- 最近随想
- 最近笔记与文章
- 本月记录天数
- 年度活跃热力图
- 最近到访地点
- 飞行总航段和总里程
- 待完善草稿
- 最近备份时间

### 7.4 默认只读与显式编辑模式

所有已经保存的内容默认以**只读展示模式**打开。页面中的正文、标题、日期、标签、地点和其他字段都不能直接点击后修改。

只有内容所有者主动点击“编辑”按钮后，页面才进入编辑模式。

要求：

- 展示页与编辑页在视觉上有明显区别。
- 展示模式不得使用可编辑输入框或 `contenteditable`。
- 编辑模式必须提供“保存”和“取消”按钮。
- 离开存在未保存修改的编辑页面时，系统必须二次提醒。
- 自动保存只在明确进入编辑模式后生效。
- 保存成功后自动退出编辑模式并返回只读展示。
- 删除、批量修改、改变可见性等操作需要独立按钮和确认步骤。
- 列表、卡片、日历、地图弹窗和详情页均遵守该规则。
- 未登录访客以及非内容所有者永远不显示编辑入口。
- 新建内容属于独立的“创建模式”，不受只读展示限制。

这样可以避免浏览、滚动、选择文字或触屏操作时误修改已有内容。

## 8. 随想模块（Thoughts）

“随想”是一个轻量短内容模块，用于保存用户想长期留下的一句话或一小段内容。

虽然模块统一称为“随想”，但内部支持两种内容来源：

1. **原创**：用户自己想到的一句话、灵感、感悟或短记。
2. **摘录**：用户从书籍、文章、视频、播客、演讲或网页中看到、听到并保存的内容。

模块名称保持简单、直观；“原创”和“摘录”作为内容类型存在，不再把模块命名为“语录”或“片语”。

### 8.1 核心功能

- 快速记录短文本，不强制填写标题。
- 创建时选择类型：
  - `原创`
  - `摘录`
- 支持标签、时间、可见性、置顶、收藏和全文搜索。
- 摘录类型支持填写：
  - 原作者或说话者
  - 来源作品名称
  - 来源类型（书籍、文章、视频、播客、演讲、网页、其他）
  - 来源链接
  - 页码、章节或时间点
  - 用户自己的补充说明
- 原创类型不强制填写来源，但可以记录产生场景或关联内容。
- 支持批量导入和导出。
- 所有已保存随想默认只读，只有点击“编辑”按钮后才能修改。

### 8.2 页面设计

- 顶部提供极简快速输入框。
- 快速输入时默认创建“原创”，也可以切换为“摘录”。
- 下方采用轻量卡片、引用列表或瀑布流展示。
- 卡片主要展示：
  - 正文
  - 原创 / 摘录类型标识
  - 作者和来源（仅摘录）
  - 标签
  - 时间
- 摘录可以使用引用视觉样式，但原创和摘录仍属于同一个“随想”模块。
- 支持按类型、来源、标签和日期筛选。

### 8.3 数据字段建议

- `thought_type`：`original` 或 `excerpt`
- `content`
- `source_author`
- `source_title`
- `source_type`
- `source_url`
- `source_locator`：页码、章节、时间点等
- `personal_note`
- `visibility`
- `created_at`
- `updated_at`

## 9. 日记模块

### 9.1 核心功能

- 按日期创建日记。
- 同一天允许发布多篇日记，不设置“一天一篇”的限制。
- 每篇日记拥有独立标题、发布时间、内容、标签和可见性。
- Markdown 编辑和预览。
- 支持图片、附件、心情、天气、地点和标签。
- 与人生轨迹双向关联。
- 支持按月、年归档和全文搜索。
- 支持私密、公开和草稿状态。
- 已保存日记默认只读，只有点击“编辑”按钮后才可修改。

### 9.2 页面设计

三栏式布局：

1. 左栏：年份、月份、日历或目录。
2. 中栏：日记列表和摘要。

## 10. 笔记与文章模块（Notes & Articles）

笔记和博客不再作为两个完全分离的模块，而是合并成统一的“笔记与文章”内容系统。

原因：

- 两者都使用 Markdown 编辑器。
- 都需要标签、分类、图片、视频、外链、搜索和版本记录。
- 将两套内容分开会产生重复的编辑器、数据结构和管理页面。
- 用户可以先写私人笔记，整理成熟后直接发布为文章，不必复制一份内容。

`/{username}/pkm`

页面内部提供以下视图：

- 标签

## 11. 文章发布与评论规则

文章评论属于“笔记与文章”模块中的文章形态，不再是独立博客模块。

- 只有已注册并登录的用户可以发表评论。
- 未登录访客可以阅读公开评论，但发表评论时应引导登录或注册。
- 作者可以为每篇文章单独开启或关闭评论。
- 普通笔记默认不开启评论。
- 评论者可以删除自己的评论。
- 文章作者可以隐藏或删除自己文章下的评论。
- 管理员只在公开评论违规、被举报或触发安全规则时进行处置。
- 评论需要限流、内容安全校验和举报入口。
- 加密空间的匿名回复不会进入普通文章评论体系。

## 12. 人生轨迹模块

### 12.1 记录内容

每一天可记录：

- 当天所在城市
- 去过的城市
- 做了什么
- 学习了什么
- 工作或项目进展
- 重要事件
- 心得
- 关联日记
- 关联片语、笔记或文章

### 12.2 多日期批量记录

人生轨迹支持一次选择多个日期并统一创建记录。

日期选择方式包括：

- 连续日期范围，例如 7 月 1 日至 7 月 5 日。
- 日历中非连续多选，例如同时选择 7 月 1 日、7 月 3 日和 7 月 8 日。
- 快捷选择工作日、周末或最近若干天。

用户填写一次地点、事项、学习内容和备注后，可以将相同内容应用到全部选中日期。

数据处理建议：

- 为每个选中日期创建独立的日轨迹记录。
- 同一批创建的记录使用 `batch_id` 关联。
- 后续编辑时提供“只修改当前日期”和“修改本批次全部记录”两个选项。
- 删除时同样提供删除单日或删除整批的选择。
- 如果某个日期已经存在相似记录，提交前提示合并、跳过或仍然新增。
- 批量创建完成后，各日期仍可单独补充不同内容。

### 12.3 展示方式

- 年度 GitHub 风格活跃热力图。
- 月历视图。
- 时间线视图。
- 支持日历多选并进入批量记录模式。
- 点击某天后显示当天所有记录和关联内容。
- 支持按地点、标签、事件类型筛选。
- 已保存轨迹默认只读，只有点击“编辑”按钮后才可修改。

### 12.4 两种轨迹记录（v1.9 新增）

v1.8 的轨迹是**按天**的：一天一条，记当天在哪、做了什么。这解决了「时间线要连续」的问题，但解决不了另一类事情——

> 「2024 年 3 月到 8 月在学吉他」「去年秋天开始尝试长跑」

这类事情**没有确切的一天**，它跨越一段时间，而且中间的每一天并不都值得单独记一条。硬塞进按天记录里，要么被拆成上百条重复内容，要么干脆记不下来。

因此轨迹分成两种，共存于同一条时间线：

| 类型 | 时间 | 用途 |
|---|---|---|
| **日轨迹** | 单个日期 | 当天在哪、做了什么（v1.8 已有） |
| **段轨迹** | 时间段（起止日期） | 一段持续的事：在学什么、在尝试什么 |

段轨迹的结束日期**可以为空**，表示「仍在进行中」。这是必要的：记录「正在学吉他」的那一刻，本来就不知道什么时候会停。

### 12.5 段轨迹的字段（v1.9 新增）

| 字段 | 必填 | 说明 |
|---|---|---|
| 起始日期 | 是 | |
| 结束日期 | 否 | 留空表示进行中 |
| 标题 | 是 | 例如「学吉他」「每周长跑」 |
| 类型 | 是 | 见下方枚举 |
| 封面图 | 否 | 走统一的图片上传，遵守第 18 章的权限与配额 |
| 详情 | 否 | Markdown，与笔记正文同一套渲染与安全规则 |
| 标签 | 否 | 与全站标签系统共用 |
| 可见性 | 是 | 公开 / 私密 / 草稿，与其他内容一致 |

**类型枚举**（初始集合，可在后台扩展）：

- `learning` 正在学习
- `attempt` 新尝试
- `habit` 习惯养成
- `project` 项目进行中
- `health` 身体与运动
- `reading` 阅读中
- `other` 其他

> 类型存**枚举值**而不是中文标签，显示时查表。理由与第 31.1 节的国家码相同：显示名会改，存进数据库的键不该跟着改。

**展示规则：**

- 热力图只统计日轨迹——段轨迹跨越几个月，计入热力图会让整片区域无差别点亮，失去信息量。
- 时间线上，段轨迹渲染为**横跨多天的色条**，日轨迹是点。
- 「进行中」的段轨迹固定显示在时间线顶部，不随日期沉下去。

### 12.6 联动规则（v1.9 修订）

- 从日记页可查看同日的全部轨迹和日记。
- 从轨迹页可打开该日期下的多篇日记。
- 当天城市记录可以自动写入足迹地图。
- 批量轨迹记录可按每个日期分别生成足迹到访记录。
- 用户可手动关闭某条轨迹对地图的影响。
- **段轨迹可以关联到行程**（第 30 章），例如「在关西那十天」既是一段轨迹也是一次行程。关联是可选的，且可解除。

## 13. 足迹与行程地图模块（v1.9 合并飞行日志）

> **v1.9 变更**：v1.8 把「足迹地图」和「飞行轨迹」拆成两章，那是**按数据类型**分的。
> 但用户看的时候是一件事：「我去过哪儿、怎么去的」。合并后首页 hero 一屏就能说完。
>
> 实现上**只有一个地图组件**，城市点与航线只是它的两个图层，共用同一个投影和同一份缩放状态。

### 13.1 地点粒度

- 国家
- 一级行政区
- 城市

地图绘制时可使用城市中心点或标准行政区坐标，但这只是展示坐标，不代表用户的精确位置。

### 13.2 核心功能

- 人生轨迹中记录当天所在城市后，可自动生成到访记录并点亮地图。
- 用户也可以不依赖人生轨迹，直接通过表单添加到访记录。
- 展示去过的国家、地区和城市。
- 统计每个城市：
  - 首次到达时间
  - 最近到达时间
  - 到访次数
  - 累计停留天数（可计算时）
  - 关联记录数量
- 鼠标悬停或点击地图区域查看统计。
- 支持世界地图和国家内部地图切换。
- 支持列表视图。
- 支持用户手动修正城市匹配结果。
- 已保存到访记录默认只读，只有点击“编辑”按钮后才可修改。

### 13.3 手动添加到访记录

足迹地图页面提供“添加到访记录”按钮。点击后打开独立表单。

建议字段：

- 国家
- 一级行政区
- 城市
- 到达日期
- 离开日期（可选）
- 到访次数（默认 1）
- 备注（可选）
- 是否关联到人生轨迹
- 可见性

表单要求：

- 国家、地区和城市采用可搜索选择器。
- 只允许保存到城市级别。
- 选择城市后自动补全国家和一级行政区。
- 没有匹配到城市时，允许提交人工校正请求或创建待确认记录，不能静默匹配到错误城市。
- 保存前检测相同城市和重叠日期的疑似重复记录。
- 用户可选择合并、跳过或仍然新增。
- 保存成功后立即更新地图和统计。

### 13.4 历史导入

支持 CSV / Excel 导入，建议字段：

- 国家
- 一级行政区
- 城市
- 到达日期
- 离开日期（可选）
- 到访次数（可选）
- 备注

## 14. 行程段模型（v1.9 由飞行日志泛化）

> **v1.9 变更**：不再建独立的「火车模块」，而是把飞行日志泛化成**行程段**：
>
> ```
> 行程段 = mode (flight / train / ferry / road) + 一组有序的停靠点
> ```
>
> - 飞行 = 两个点（起飞、落地），中转即两段
> - 火车 = N 个点（实际经过的站）
>
> 飞机与火车共用同一张表、同一套统计、同一个渲染层。
> 将来加轮渡、自驾、长途大巴只是加一个枚举值，不是加一个模块。
>
> **地图渲染已并入第 13 章**，本章只保留数据模型与录入规则。
> 详细取舍见第 31.2 节（建议写成 ADR）。

### 14.0 行程段字段（v1.9）

| 字段 | 说明 |
|---|---|
| `mode` | `flight` / `train` / `ferry` / `road` |
| `stops` | **有序**停靠点数组，至少两个 |
| 日期、出发与到达时间 | |
| 里程 | 飞行可由坐标算出；陆路按实际里程录入 |
| 承运方与班次号 | 航空公司 + 航班号 / 铁路公司 + 车次 |
| 舱位、票号、票状态、机型 | 飞行专有，其他 mode 留空 |
| 备注、数据来源 | |

**停靠点的粒度按 mode 区分：**

- **飞行**：机场（IATA/ICAO），因为同城多机场是有意义的区别（PVG / SHA）。
- **陆路**：**城市即可**。全球没有像 OurAirports 那样干净的火车站数据集，
  而个人记录用城市粒度完全够用——「上海 → 南京 → 北京」，没人需要知道是虹桥还是上海站。
  这省掉了一整个数据源。

**顺带的好处**：录停靠点的同时也就记下了「这趟经过哪些城市」，本身就是足迹数据，
不需要再单独录一遍。

**唯一实质差别：陆路贴地面走。** 大圆弧对飞机物理正确，对火车完全错误——
上海到北京画一道向北弓起的弧线，懂地理的人一眼就看出不对。
陆路用经过站点连成的**折线**并稍作平滑，不要用 `geoInterpolate`。渲染细节见第 32.6 节。


### 14.1 导入字段

必须支持以下字段：

- 日期
- 航空公司
- 航班号
- 出发城市
- 出发时间
- 到达城市
- 到达时间
- 里程数
- 客票号
- 客票状态

建议增加：

- 出发机场 IATA
- 到达机场 IATA
- 舱位
- 机型
- 备注
- 数据来源

### 14.2 去重规则

基础唯一键：

`日期 + 航班号 + 出发时间 + 到达时间`

为减少格式差异造成的重复，导入前应统一：

- 日期格式
- 时区
- 航班号大小写和空格
- 城市或机场名称
- 时间精度

发生疑似重复时，支持三种处理：

- 跳过
- 覆盖
- 保留并标记冲突

### 14.3 坐标匹配

- 优先使用 IATA / ICAO 机场代码匹配。
- 仅有城市名时，进入人工确认或候选机场选择流程。
- 预置全球机场坐标库。
- 保留用户修正后的映射结果。

### 14.4 可视化（v1.9：渲染已并入第 13 章地图）

> 世界地图弧线、机场点位、图层与缩放交互全部由第 13 章的**单一地图组件**承担，
> 技术方案见第 32 章。本节只保留筛选与统计的需求。

- 按年份、mode、承运方和出发地筛选。
- 支持动画飞线和静态模式。
- 统计（v1.9 起按 mode 分组，不再只统计飞行）：
  - 总行程段（按 mode 分别统计）
  - 总里程
  - 总在途时间
  - 去过的机场、城市和国家数
  - 最常乘坐的航司 / 铁路公司
  - 最常用机场 / 车站城市
  - 年度出行趋势

### 14.5 行程段表格

- 排序、筛选、分页。
- 手动新增、编辑、删除。
- 批量导入和导出。
- 标记异常、取消、退票或未知状态。

## 15. 多入口密码加密互动空间

### 15.1 页面定位

独立加密互动功能调整为**多个相互隔离的加密空间**。

每个用户拥有自己的统一空间入口，例如：

`/{username}/space`

站点主人还可以选择将根路径 `/space` 作为 `/{owner_username}/space` 的别名。

页面只显示欢迎语和密码输入框。访客输入不同的空间密码后，系统自动识别并进入与该密码绑定的空间。

示例：

- 输入密码 A → 进入“空间 A”
- 输入密码 B → 进入“空间 B”
- 输入密码 C → 进入“空间 C”

访客不需要先选择空间，也不需要知道空间名称或空间地址。空间密码同时承担“识别目标空间”和“验证访问权限”两项作用。

### 15.2 主人登录态

站点主人登录后：

- 可在后台查看自己创建的全部加密空间。
- 可在管理员为该账号配置的空间数量上限内新建空间。
- 可编辑、停用和删除空间。
- 可为每个空间设置独立名称、说明、封面、访问密码和会话有效期。
- 可直接进入任意空间，无需再次输入空间密码。
- 可在指定空间中发布标题、时间、Markdown 正文、图片和附件。
- 可查看、回复、隐藏或删除该空间内的全部访客回复。
- 主人的回复显示“作者”或“空间主人”标识。
- 可单独关闭某个空间的访客回复功能。
- 可单独更换某个空间的访问密码；更换后可选择立即使旧访客会话失效。

### 15.3 未登录访客态

第一步：

- 进入目标用户的统一空间入口 `/{username}/space`。
- 页面内容完全隐藏。
- 只显示欢迎语、密码输入框和进入按钮。

密码验证成功后：

- 后端识别该密码对应的空间。
- 为访客签发仅对该空间有效的短期访问会话。
- 自动跳转或展示该空间的时间线。
- 访客只能查看当前密码对应的空间。
- 访客不能通过当前会话访问其他空间。
- 访客无权发布主内容。
- 访客可对当前空间内的内容回复。
- 昵称不是必填项；不填写时按匿名回复处理。

密码错误时：

- 不提示“空间不存在”还是“密码错误”，统一返回验证失败，避免泄露空间信息。
- 连续失败触发限流、临时锁定或验证码。

### 15.4 空间隔离规则

每个空间必须拥有独立的：

- 空间 ID
- 空间名称
- 访问密码
- 内容时间线
- 图片与附件
- 回复
- 访客会话
- 回复开关
- 会话有效期
- 启用或停用状态

所有空间内容和回复都必须关联 `space_id`。

一个空间的访客会话不能读取另一个空间的数据。即使多个空间属于同一个主人，也必须在后端逐次校验会话中的 `space_id`。


空间数量采用配额控制：

- 平台设置一个默认空间上限。
- 管理员可以按用户单独提高或降低 `encrypted_space_limit`。
- 用户界面显示“已使用数量 / 当前上限”。
- 降低上限后不自动删除已有空间；如果已有数量达到或超过新上限，只禁止继续创建。
- 停用空间仍计入上限，彻底删除后才释放名额。

### 15.5 密码映射与安全实现

由于密码需要决定进入哪个空间，后端不能只保存普通密码哈希后逐个暴力比对，也不能保存明文密码。

建议采用两层处理：

1. 使用服务器密钥对规范化后的密码计算不可逆检索指纹，例如 HMAC，用于快速定位候选空间。
2. 对该空间保存独立的 Argon2id 密码哈希，用于最终验证。

数据库可保存：

- `password_lookup_fingerprint`
- `password_hash`
- `password_changed_at`

要求：

- 同一站点内用于进入空间的密码必须唯一。
- 创建或修改密码时检查冲突。
- 不保存或记录密码明文。
- 日志中不得出现访客提交的密码。
- 所有验证请求必须通过 HTTPS。
- 空间密码应支持最小长度和强度要求。

### 15.6 访客会话

验证成功后，建议通过 `HttpOnly + Secure + SameSite` Cookie 保存短期会话。

为了支持同一访客同时在不同浏览器标签中打开多个空间，验证成功后可以进入：

`/{username}/space/{opaque_space_key}`

其中 `opaque_space_key` 是不可猜测的随机标识，不直接暴露空间名称。后端仍必须同时验证访客会话与 `space_id`，不能只凭 URL 放行。

会话至少包含：

- 访客会话 ID
- 对应 `space_id`
- 签发时间
- 过期时间
- 可选的设备或风险信息

会话只能用于：

- 获取当前空间内容
- 获取当前空间回复
- 在当前空间提交回复

会话不能用于：

- 进入其他空间
- 发布主内容
- 访问用户后台
- 读取普通私密内容

### 15.7 内容展示

每个空间采用独立时间线，默认按时间倒序。

时间线主页卡片显示：

- 封面或首图
- 时间
- 标题
- 摘要

详情页顺序：

1. 标题
2. 居中显示时间
3. 按 Markdown 原始结构渲染正文
4. 图片和附件
5. 回复列表与回复框

主人可为不同空间设置不同的：

- 空间名称
- 欢迎语
- 封面
- 页面主题
- 是否显示已填写的昵称
- 是否允许回复
- 是否显示内容总数

### 15.8 推荐数据模型

建议增加或调整以下实体：

- `encrypted_spaces`
  - `id`
  - `user_id`
  - `name`
  - `description`
  - `cover_image`
  - `password_lookup_fingerprint`

### 15.9 备份与恢复要求

- 各空间回复

密码不得以明文写入导出包。恢复后应保留原密码哈希，或要求主人重新设置空间密码。


## 16. 搜索、分类与组织

- 全文搜索标题和正文。
- 按内容类型筛选。
- 按标签、分类、文件夹、日期、地点和可见性筛选。

- 私密内容不得出现在游客搜索或搜索引擎索引中。

建议统一标签体系；日记保留自己的归档方式，笔记与文章共享内容系统，但可分别使用文件夹、分类和系列。

## 17. 页脚运行时长与访问统计

### 17.1 页脚展示

网站所有公开页面底部统一展示站点状态信息。

推荐格式：

`本站已运行 123 天 08 小时 26 分 41 秒 · 总访问量 12,345 次 · 独立访客 3,210 人`

最低必须展示：

- 系统上线运行时长
- 总访问量（PV）

建议同时展示：

- 独立访客数（UV）
- 今日访问量（可选）
- 数据更新时间或统计说明入口（可选）

页脚应适配桌面端和移动端，不应影响正文阅读。

### 17.2 运行时长定义

这里展示的应是**站点首次正式上线后的累计时长**，而不是当前后端进程的启动时长。

系统保存一个持久化字段：

- `site_launched_at`：站点首次正式上线时间
- `site_timezone`：站点展示时区

要求：

- 服务器重启、容器重建、版本更新或迁移后，运行时长不能归零。
- `site_launched_at` 纳入系统备份和恢复。
- 只有超级管理员可以修改该时间。
- 修改操作必须写入审计日志。
- 天数不限制为三位数，超过 999 天后继续正常增长。
- 小时、分钟和秒固定补足两位。

显示格式：

`{累计天数} 天 {小时:02d} 小时 {分钟:02d} 分 {秒:02d} 秒`

### 17.3 实时增长机制

后端返回：

- `site_launched_at`
- `server_now`

前端根据两者计算初始时长，并使用本地定时器每秒增加一次。

为避免用户设备时间不准确：

- 初始时间必须以服务器时间为准。
- 页面重新获得焦点、网络恢复或运行一段时间后，应重新向服务器校时。
- 浏览器标签页休眠恢复后，不能只继续累加旧计时器，而应重新计算真实差值。
- JavaScript 被禁用时，可退化为显示页面生成时的静态时长。

### 17.4 访问量指标定义

建议至少记录以下两类指标：

#### 页面浏览量（PV）

一次有效页面打开计为一次页面浏览。

只统计 HTML 页面访问，不统计：

- API 请求
- 图片、CSS、JavaScript 和字体等静态资源
- 健康检查
- 后台定时任务
- 已识别的搜索引擎爬虫和恶意机器人

#### 独立访客（UV）

在一个统计周期内，同一访客只计算一次。

建议通过匿名访客 Cookie 或短期匿名标识统计，不长期保存原始 IP 地址。

可提供：

- 今日 UV
- 本月 UV
- 累计 UV

累计 UV 是估算值，应在统计说明中注明其口径。

### 17.5 防止重复和刷量

建议规则：

- 同一匿名会话在短时间内反复刷新同一页面，只计一次有效访问。
- 默认去重窗口可设为 30 分钟。
- 站点主人和管理员登录后的访问，可在后台配置是否计入。
- 对异常高频访问进行限流和机器人过滤。
- 不允许前端直接提交任意访问量增量；计数必须由后端根据真实页面请求或统计事件完成。
- 统计写入可以异步聚合，避免每次访问都对主数据表产生高频锁竞争。

### 17.6 平台级与用户级统计

系统开放多用户后，需要区分两层统计。

#### 平台级统计

统计整个 `life.555978.xyz`：

- 全站累计 PV
- 全站累计 UV
- 今日、昨日、本月访问趋势
- 各用户主页访问排行

平台级详细数据仅管理员可见。

根主页 `/` 的页脚可以展示全站统计，或按后台配置展示站点主人主页统计。

#### 用户级统计

每个用户单独统计：

- `/{username}` 及其公开子页面的 PV
- 该用户公开主页的 UV
- 各模块和各公开文章的访问量

例如访问 `/euan/blog/article-a`，既计入：

- 平台总访问量
- `euan` 的用户级访问量
- 对应文章的访问量

其他用户访问 `/alice/...` 时，不应计入 `euan` 的个人主页统计。

### 17.7 页脚展示口径建议

推荐默认规则：

- `/`：展示“站点运行时长 + 全站 PV + 全站 UV”。
- `/{username}` 及其公开子页面：展示“站点运行时长 + 当前用户主页 PV + 当前用户主页 UV”。
- 登录后的私人工作台 `/app`：可以不展示公开访问量，只展示系统运行状态。
- 管理后台 `/admin`：展示完整统计看板，不使用简化页脚数字代替分析数据。

用户可以在个人设置中选择：

- 是否公开展示自己的主页访问量
- 只展示 PV，还是同时展示 UV
- 是否显示今日访问量
- 数字精确显示，还是使用 `1.2 万` 等简写

### 17.8 数据结构建议

建议增加：

- `site_settings`
  - `site_launched_at`
  - `site_timezone`
  - `count_owner_visits`
  - `analytics_enabled`

- `page_view_events`（可选的短期原始事件表）
  - `id`
  - `user_id`
  - `content_id`
  - `route`
  - `anonymous_visitor_id`
  - `session_id`
  - `occurred_at`
  - `is_bot`
  - `referrer_domain`

- `visit_statistics_daily`
  - `date`
  - `scope_type`
  - `scope_id`
  - `pv`
  - `uv`

- `visit_statistics_total`
  - `scope_type`
  - `scope_id`
  - `pv`
  - `uv`
  - `updated_at`

其中 `scope_type` 可包含：

- `platform`
- `user`
- `content`
- `module`

### 17.9 隐私要求

- 默认不长期保存完整 IP 地址。
- 如为安全分析临时处理 IP，应使用严格保留期限和访问权限。
- 不在公开页面展示访客身份、IP、设备或精确位置。
- Referrer 建议只保存来源域名，不保存可能包含隐私参数的完整 URL。
- 用户关闭统计后，不再采集其个人主页分析事件；平台为安全和容量管理保留的最低必要日志应在隐私政策中说明。

### 17.10 备份与恢复

备份必须包含：

- `site_launched_at`
- 累计访问统计
- 每日聚合统计
- 用户级和内容级统计
- 统计配置

恢复后：

- 站点运行时长继续按照原上线时间计算。
- 累计访问量不得归零。
- 不要求长期原始访问事件全部保留；每日和累计聚合数据必须保留。

## 18. 图片、小视频与外部链接

### 18.1 允许的内容类型

系统只允许以下媒体和链接形式：

1. **图片上传**
2. **小视频上传**
3. **站外 HTTPS 链接**
4. **受支持视频平台的嵌入链接**

暂时不允许上传其他附件，包括但不限于：

- ZIP、RAR、7Z 等压缩包
- EXE、APK、DMG、MSI 等可执行或安装文件
- Python、Shell、JavaScript 等代码文件
- Word、Excel、PowerPoint、PDF 等文档
- 数据库文件
- 字体文件
- 音频文件
- 未明确允许的其他二进制文件

笔记或文章中需要分享代码、程序、工具、安装包、数据集或文档时，只能使用站外网盘、代码托管平台或项目官网链接，不允许把这些文件直接上传到本系统。

### 18.2 图片上传

支持在随想、日记、笔记与文章、人生轨迹和独立加密空间内容中插入图片。

要求：

- 支持拖拽、粘贴和文件选择。
- 支持常用安全图片格式，如 JPEG、PNG、WebP 和 GIF。
- SVG 默认不允许直接上传；如未来开放，必须经过严格清洗。
- 服务端必须校验真实文件类型，不能只检查文件扩展名。
- 自动生成缩略图并进行适度压缩。
- 支持图片说明和替代文本。
- 图片权限继承所属内容的权限。
- 私密图片不能通过公开静态 URL 绕过鉴权访问。
- 已保存内容仍然默认只读，只有进入编辑模式后才能增加、替换或删除图片。

### 18.3 小视频上传

用户可以上传不超过 **20 MB** 的小视频，并插入到内容正文中。

建议首期只允许浏览器兼容性较好的格式：

- MP4
- WebM

具体要求：

- 单个视频文件最大 20 MB，以服务端校验结果为准。
- 前端选择文件时立即提示大小和格式是否符合要求。
- 服务端再次校验文件大小、MIME 类型和文件头。
- 文件名由系统重新生成，不能直接使用用户原始文件名作为存储路径。
- 视频文件存放在用户隔离目录或对象存储的用户前缀下。
- 视频不得被当作可执行文件处理。
- 视频权限继承所属内容的权限。
- 私密视频必须经过后端鉴权或短期签名地址访问。
- 支持在线播放、暂停、音量、全屏和移动端播放。
- 默认不自动播放，避免流量消耗和干扰。
- 可以生成封面图；生成失败时使用系统默认封面。
- 删除正文中的视频引用时，不立即物理删除文件；由资源清理任务确认无其他内容引用后再删除。
- 用户视频总容量应计入存储配额，管理员可以调整每个用户的存储上限。

### 18.4 外部链接

Markdown 编辑器和内容编辑页面允许插入站外链接。

典型用途：

- Google Drive 等公开网盘分享链接
- 哔哩哔哩视频链接
- YouTube 视频链接
- GitHub、GitLab 或其他代码托管链接
- 项目官网、软件下载页、文档页
- 其他合法的公开 HTTPS 页面

要求：

- 只允许 `https://` 链接；首期不允许 `javascript:`、`data:`、`file:`、自定义协议和普通 HTTP。
- 用户可以为链接设置显示标题和说明。
- 普通链接以安全链接卡片或文本链接展示。
- 外部链接默认在新标签页打开，并设置 `noopener`、`noreferrer` 等安全属性。
- 页面应明确标识“外部链接”，提醒访问者内容由第三方网站提供。
- 系统不自动下载、镜像或备份第三方链接中的文件。
- 外链失效、权限变化或第三方删除内容时，本系统无法保证继续访问。
- 可选提供外链状态检查，但不得绕过第三方访问权限。

### 18.5 视频平台链接与嵌入

对于哔哩哔哩、YouTube 等受支持平台的视频链接，系统可以将链接转换为站内播放器或预览卡片。

实现要求：

- 使用受控的平台白名单和官方允许的嵌入方式。
- 不允许用户直接提交任意 `<iframe>`、`<script>` 或 HTML 嵌入代码。
- 后端解析并规范化视频链接，只保存原始 URL、平台类型和可验证的视频标识。
- 无法安全识别或不支持嵌入时，退化为普通外部链接卡片。
- 播放器使用懒加载，避免页面加载时同时请求大量第三方资源。
- 对第三方 Cookie、追踪和隐私影响在隐私政策中说明。
- 用户可以选择“嵌入播放器”或“只显示链接卡片”。
- 视频作者删除、设为私密或限制地区后，嵌入内容可能无法播放。

### 18.6 Google Drive 等网盘链接

网盘链接主要用于分享：

- 笔记或文章相关代码
- 工具或安装包
- 数据文件
- 文档
- 大于 20 MB、不适合上传到本站的视频
- 其他本站禁止直接上传的附件

系统只保存链接和用户填写的描述，不保存网盘中的文件副本。

展示建议：

- 文件或资源名称
- 来源平台
- 用户填写的说明
- “打开外部链接”按钮
- 可选的外部链接风险提示

如果第三方平台允许安全预览，可以显示预览卡片；否则只显示链接，不尝试绕过其分享权限。

### 18.7 Markdown 与编辑器交互

编辑器提供统一的“插入媒体”菜单：

- 上传图片
- 上传小视频
- 插入外部链接
- 插入视频平台链接

插入后在 Markdown 中保存为结构化语法或受控扩展节点，避免保存任意危险 HTML。

要求：

- 编辑模式中显示预览。
- 展示模式只负责渲染，不能直接替换链接或媒体。
- 复制粘贴 URL 时，可识别受支持平台并询问使用普通链接还是嵌入播放器。
- 所有 Markdown、URL 和嵌入结果都必须经过服务端安全清洗。
- 导出时保留原始外链 URL、标题、说明和平台信息。

### 18.8 安全控制

上传和外链功能必须执行以下限制：

- 使用上传类型白名单，不采用黑名单。
- 校验扩展名、MIME 类型和文件特征。
- 限制单文件大小、用户总容量和上传频率。
- 上传文件与应用程序代码目录隔离。
- 文件存储目录禁止执行。
- 对上传失败、类型不符和超限行为记录安全日志。
- 不允许通过重命名把压缩包、脚本或可执行文件伪装成图片或视频。
- 外部 URL 在保存前进行格式校验。
- 后端如需获取外链元数据，必须防范 SSRF，不允许访问内网、环回地址、云主机元数据地址和非公开 IP。
- 不渲染用户提交的任意 iframe、script、object、embed 或事件处理属性。
- 管理员可以全局关闭视频上传、第三方嵌入或特定外链平台。

建议增加或明确：

- `media_assets`
  - `id`
  - `user_id`
  - `media_type`：`image` 或 `video`
  - `storage_key`
  - `original_filename`
  - `mime_type`

  - `checksum`

  - `height`
  - `duration_seconds`
  - `thumbnail_key`
  - `created_at`
  - `deleted_at`

- `external_links`
  - `id`
  - `user_id`
  - `content_id`

内容与媒体之间建议通过关联表维护，便于识别同一资源是否被多个内容引用。

### 18.10 备份与恢复

用户级导出和系统级备份必须包含：

- 本地上传的图片
- 本地上传的小视频

- 外部链接 URL
- 链接标题、说明、平台和嵌入设置

外部链接指向的第三方文件或视频不纳入本站备份。恢复后只恢复链接本身，不能保证第三方资源仍然有效。

## 19. 数据导出、备份与恢复

需要区分用户级数据迁移与管理员级系统灾难恢复。

### 19.1 用户级可迁移导出与合并恢复

每个用户可导出自己的数据包，建议包含：

- `manifest.json`：版本、用户、导出时间和数据结构信息
- Markdown 或 JSON 内容数据
- 地点与飞行记录 CSV
- 本地上传的图片和小视频
- 外部链接及嵌入信息
- 标签、目录、关联关系
- 独立空间内容与回复
- 可见性和时间信息

用户将数据包导回账号时，**只支持合并模式，不提供覆盖现有账号数据的模式**。

合并规则：

- 导入前展示新增、重复、冲突和错误数量。
- 具有相同稳定 UUID 且内容完全相同的数据直接跳过。
- UUID 相同但内容不同的数据不覆盖现有内容，而是创建冲突副本并提示用户处理。
- 没有 UUID 的旧格式数据使用类型、时间、标题和内容摘要进行疑似重复检测。
- 标签、文件夹、城市和机场等基础数据优先复用已有记录。
- 图片和附件通过校验值去重。
- 独立空间密码哈希和空间关联必须正确迁移，发生冲突时要求用户重新确认或设置密码。
- 导入完成后提供详细报告和冲突处理入口。
- 合并导入失败时必须回滚本次导入，不影响原有数据。

目标：用户可以多次导入历史备份，而不会意外覆盖或清空当前数据。

### 19.2 系统级完整备份

管理员备份整个服务器端系统：

- 数据库
- 本地图片和小视频
- 外部链接元数据
- 系统配置
- 用户与权限
- 任务状态
- 备份校验文件

个人阶段的数据虽然存储在服务器上，也不能只依赖服务器单盘。至少应保留一份服务器外部备份，例如另一台服务器、对象存储或用户定期下载的加密备份包。

备份包应：

- 包含版本号和校验值
- 支持加密
- 支持下载
- 支持定期自动备份
- 支持恢复前校验和预演

### 19.3 系统灾难恢复

管理员级完整系统恢复用于新服务器、灾难故障或完整迁移，与普通用户的“合并导入”不同。

流程：

1. 上传或选择系统备份。
2. 校验格式、版本和完整性。
3. 进入维护模式并停止写入。
4. 自动创建恢复前快照。
5. 展示恢复范围和风险。
6. 执行完整恢复和数据库迁移。
7. 运行完整性检查。
8. 输出恢复报告并恢复服务。

不建议在系统正常运行并持续写入时直接覆盖数据库文件。

## 20. 多用户数据隔离

从第一版开始，所有用户数据表都必须带 `user_id` 或等价所有权字段。

后端每次读写都必须同时校验：

- 当前会话身份
- 资源所有者
- 内容可见性
- 角色权限

禁止仅由前端传入 `user_id` 决定数据归属。

建议：

- 用户上传目录按用户隔离。
- 导出任务按用户隔离。
- 搜索索引按权限过滤。
- 缓存键包含用户和可见性信息。
- 管理员操作留审计日志。

- 系统公告
- 服务条款和隐私政策配置

### 21.1 管理员访问用户私密数据的原则

推荐采用“默认不可见、必要时破窗访问（Break-glass）”机制。

正常情况下：

- 管理后台不提供浏览用户私密日记、片语、笔记与文章、轨迹或加密空间正文的入口。
- 管理员只能查看账号状态、存储用量、内容数量、错误状态等必要元数据。
- 管理员不能通过“模拟登录”直接进入用户账号。
- 管理员不能通过常规后台批量导出用户私密数据。
- 对公开内容和公开评论的管理不需要访问用户私密数据。

只有以下情况可以申请临时访问：

1. 用户主动发起支持请求，并明确授权查看指定内容。
2. 严重安全事件，需要确认账号是否被入侵或系统是否遭到利用。
3. 有效且必须执行的法律要求。
4. 数据恢复或损坏修复确实无法仅通过元数据完成。

破窗访问要求：

- 必须填写具体原因、工单编号和访问范围。
- 默认只读，并限制到指定用户、指定模块或指定记录。
- 设置很短的有效期，到期自动撤销。
- 如果系统有多名管理员，建议由第二名管理员审批。
- 单管理员阶段至少进行二次确认，并向管理员邮箱发送安全通知。
- 全部查看、搜索、下载和导出行为写入不可随意删除的审计日志。
- 除法律禁止通知外，应在访问前或访问后通知用户。
- 用户可以在账号安全页查看与自己相关的管理员访问记录。
- 访问结束后自动关闭权限并生成访问报告。

技术边界：

由于服务器需要处理搜索、Markdown 渲染、备份和跨设备同步，普通私密数据在服务端通常不是端到端加密。服务器所有者在技术上仍可能直接接触数据库。因此当前阶段应依靠权限隔离、审批、审计和制度限制。

如果未来要求“包括服务器管理员也绝对无法读取”，需要另行设计客户端端到端加密的私密保险箱模式，但它会明显限制全文搜索、服务端处理、恢复和多设备同步体验。

## 23. 技术架构建议

### 23.1 前端

推荐：

- Vue 3 + Vite，或 React / Next.js
- TypeScript
- Tailwind CSS 或成熟 UI 组件库
- Markdown 编辑器：Vditor、Milkdown 或同类组件
- 地图：Apache ECharts、MapLibre GL / Mapbox GL
- 数据表格：支持大数据分页、筛选和字段映射

### 23.2 后端

推荐：

- FastAPI、NestJS 或 Go
- REST API 起步，后续可按需增加 WebSocket
- 后台任务处理导入、备份、图片处理和邮件发送
- 使用 ORM 和迁移工具，避免数据库结构难以升级

### 23.3 数据库与存储

个人版仍然采用服务器端存储，可采用：

- 部署在服务器上的 SQLite 数据库
- 服务器磁盘或挂载数据盘中的文件存储
- 所有设备通过同一套后端 API 读写数据

这里的“本地文件存储”是指文件保存在服务器本地磁盘，而不是保存在用户当前使用的电脑或手机中。

公众多用户版建议升级为：

- PostgreSQL
- S3 兼容对象存储
- Redis（缓存、限流、短期会话或任务队列，可选）


当前版本和可预见的公众版都只使用平台主域名与 `/{username}` 路由，暂不支持用户绑定自定义域名。

为了平滑迁移，应从第一天就抽象：

- 数据访问层
- 文件存储接口
- 备份格式
- 数据导入导出格式

### 23.4 部署

个人阶段也应部署到持续在线的服务器，例如 VPS、云服务器、家用服务器配合安全公网访问，或其他稳定托管环境。

建议提供：

- Docker Compose
- 独立持久化数据卷
- 环境变量配置
- 域名与 HTTPS
- 自动数据库迁移
- 健康检查
- 日志轮转
- 定期备份
- 异地或对象存储备份副本
- 反向代理
- 移动端响应式页面
- 可选 PWA，以便在手机桌面快速打开

设备端不保存权威数据。服务器数据库和文件存储是唯一可信数据源，用户在不同设备登录后看到相同内容。

## 25. 非功能需求

### 25.1 性能

- 普通内容列表首屏应快速加载。

- 大批量导入必须异步处理并显示进度。

- 桌面端和移动端均可使用。

- 网络中断后不应轻易丢失草稿。

- 全站 HTTPS。

- 私密附件不能通过静态目录直接公开。
- 日志不得记录密码、Token 或完整敏感数据。
- 定期更新依赖并检查漏洞。

### 25.4 可维护性

- 数据库迁移可回滚。
- 关键业务有自动化测试。
- 导入和恢复有版本兼容策略。
- 提供错误日志和可观测性。

## 27. 验收标准示例

### 注册登录

- 用户可以通过邮箱完成注册、验证、登录、退出和密码重置。
- 关闭注册时，普通访客不能创建账号。
- 邀请注册时，无效邀请码不能注册。
- 用户 A 无法读取或修改用户 B 的私密数据。

### 内容权限与编辑安全

- 私密内容无论通过列表、详情、搜索还是附件地址都不能被游客访问。
- 公开内容可免登录访问。
- 普通内容没有单条密码保护，密码只用于独立加密空间。
- 已保存内容打开后默认只读，点击正文、标题、标签或日期不会直接修改数据。
- 只有内容所有者点击“编辑”后才进入编辑模式。
- 编辑未保存时离开页面会收到提醒。

### 备份恢复

- 导出包包含文字、图片、附件、标签、轨迹、飞行记录、独立空间及回复。
- 用户导入数据时只进行合并，不覆盖或清空现有内容。
- 重复数据被跳过，冲突数据生成待处理副本。
- 系统级灾难恢复可以在新部署中完整恢复全部数据。
- 失败导入或恢复不会破坏原有数据。

### 飞行导入

- 重复导入相同航班不会产生重复记录。
- 无法自动匹配机场时会提示用户处理，而不是静默写入错误坐标。
- 导入前可预览，导入后有成功、跳过和错误统计。


### 日记、轨迹与足迹

- 同一天可以创建和发布多篇日记。
- 人生轨迹可以一次选择连续或非连续的多个日期并批量创建。
- 批量轨迹可以选择只编辑单日或编辑整批。
- 用户可以直接在足迹地图页面通过表单添加城市级到访记录。
- 系统不会保存用户的精确 GPS 位置。


### 图片、视频与外链

- 允许上传图片和不超过 20 MB 的 MP4/WebM 小视频。
- 超过 20 MB 或不在白名单中的视频会被前端提示并由服务端拒绝。
- ZIP、可执行文件、代码文件、Office 文档、PDF 和其他普通附件不能上传。
- 笔记与文章中的代码、工具、安装包和文档通过站外链接分享。
- 哔哩哔哩和 YouTube 等受支持链接可以安全嵌入；不支持时退化为普通链接。
- 用户提交任意 iframe、script 或危险 URL 协议时，系统拒绝或安全清洗。
- 私密内容中的本地视频不能通过公开地址绕过权限访问。
- 备份恢复后，本地图片和小视频保持可用，外部链接及其说明保持一致。

### 评论与空间配额

- 未登录访客不能发布普通文章评论。
- 已登录注册用户可以在作者开启评论后发表评论。
- 加密空间访客回复不强制填写昵称。
- 每个用户的加密空间数量受配额限制，管理员可单独调整。


### 运行时长与访问统计

- 服务重启、容器更新和重新部署后，页脚运行时长不会归零。
- 页脚秒数实时增长，浏览器休眠恢复后能够自动校正。
- 静态资源和 API 请求不会被计入页面访问量。
- 同一访客短时间重复刷新不会造成明显虚增。
- 访问某个用户页面时，平台统计和该用户统计均正确增加，但不会增加其他用户的统计。
- 备份恢复后，上线时间和累计统计保持一致。

## 28. 已确认的产品决策与管理原则

1. **日记数量**：同一天允许发布多篇日记，每篇独立管理。
2. **密码保护范围**：普通内容不使用密码保护；密码只用于整个独立加密空间。
3. **空间回复昵称**：昵称不强制，访客可以匿名回复。
4. **加密空间数量**：每个用户设置空间数量上限；平台有默认值，管理员可以按用户调整。
5. **足迹位置精度**：只记录到城市级别，不保存精确地址或 GPS 轨迹。
6. **跨用户内容展示**：每个用户主页及其子页面只展示该用户自己的内容，不混入其他用户的推荐、搜索结果或热门内容。当前不建设全平台内容聚合页。
7. **普通博客评论**：只有注册并登录的用户可以发表评论；未登录访客只能阅读。
8. **用户数据恢复**：只采用合并模式，不覆盖现有账号数据；重复跳过，冲突生成副本。
9. **自定义域名**：暂不允许，所有用户统一使用 `life.555978.xyz/{username}`。
10. **管理员访问私密数据**：默认不可见。只有用户授权支持、严重安全事件、必要法律要求或无法通过元数据完成的数据修复时，才允许限时、限范围、默认只读的破窗访问，并完整审计和通知用户。
11. **内容编辑方式**：所有已保存内容默认只读，只有点击明确的“编辑”按钮后才能修改。
12. **人生轨迹批量记录**：支持一次选择多个连续或非连续日期进行统一记录。
13. **足迹手动录入**：除人生轨迹自动生成和表格导入外，用户可以直接通过表单添加城市级到访记录。
14. **媒体与附件策略**：允许图片和单个不超过 20 MB 的小视频；其他附件禁止上传。代码、工具、安装包、文档和大文件统一使用站外公开链接分享。
15. **视频外链**：支持哔哩哔哩、YouTube 等受控平台链接嵌入，也支持 Google Drive 等网盘分享链接；外部资源不复制到本站。
16. **随想模块命名**：原“语录”模块更名为“片语（Thoughts）”，同时支持用户原创随想和外部摘录；摘录可记录作者、作品、链接、页码或时间点。
17. **笔记与博客合并**：笔记和技术博客合并为“笔记与文章（Notes & Articles）”，统一使用 `/{username}/pkm` 内容入口（实际路由不含空格）；通过 `content_kind` 区分笔记与文章。笔记可直接发布为文章，不复制正文。

## 30. 旅行记录 / 行程模块（v1.9 新增）

### 30.1 解决的问题

当前旅行数据是散的：

- 航段属于飞行日志
- 城市属于足迹
- 感想属于日记
- 照片无处安放

但真实的心理单位不是「某一个航段」，而是**「2024 年秋天那趟关西」**。缺了这一层，感悟和照片确实无处可放。

### 30.2 行程是一个容器，不是新数据

**行程本身不存旅行的原始数据**，它只把已有记录归拢起来：

| 字段 | 说明 |
|---|---|
| 标题 | 「2024 秋・关西」 |
| **日期区间** | 起止日期，不是单个日期 |
| 封面图 | |
| 正文 | 整趟的感悟，Markdown |
| 图集 | |
| 引用 | 包含哪些行程段、经过哪些城市、关联哪些日记与随想 |

> **这与笔记模块的「系列 > 文件夹 > 内容」是同一个概念**——行程就是旅行记录的系列。
> 已经写过一遍的多对多归属逻辑可以直接搬，包括「内容可属于多个容器」和「继承关系不重复计入」这两条规则。

### 30.3 内容归属分层（避免重复录入）

用户最容易困惑的是「这件事该记在哪」。规则要简单到不用想：

| 想记的内容 | 放哪 |
|---|---|
| 当天的心情、发生了什么 | **日记**（有日期，自动上时间线） |
| 一个瞬间、一句话 | **随想** |
| 整趟行程的感悟 + 图集 | **行程本身** |
| 想公开发的深度游记 / 攻略 | **笔记与文章**，关联到行程 |

**同一段文字不应该录两遍。** 行程页展示日记时是**引用**，不是复制；编辑日记，行程页跟着变。

### 30.4 关联与解除（映射表）

行程与其他内容是**多对多关联**，存在通用关联表里（第 31.4 节）。

**自动建议，人工确认：**

| 关联对象 | 建议依据 | 是否自动生效 |
|---|---|---|
| 行程段 | 日期落在行程区间内 | 建议，需确认 |
| 城市 | 由关联的行程段的停靠点推导 | 自动 |
| 日记 | 日期落在行程区间内 | **建议，需确认** |
| 随想 | 日期落在行程区间内 | **建议，需确认** |
| 笔记与文章 | 无自动依据 | 手动关联 |

**为什么日记和随想必须人工确认：** 旅行期间写的东西不一定和旅行有关。在关西的第三天写了一篇关于工作的日记，它不属于这趟行程。**自动关联会把它错误地放进游记里，而用户未必会注意到。**

**因此必须支持解除关联，且解除要满足：**

- 在**行程页**可以直接把某条日记 / 随想移出本行程。
- 在**日记 / 随想自身的页面**也能看到「属于哪个行程」并解除——用户不一定是从行程页发现问题的。
- 解除只断开关联，**不删除内容本身**，也不影响它在日记时间线上的位置。
- 解除后不应被下一次自动建议重新加回来：需要记住「这条被人工排除过」。

> 最后一条容易被忽略，但缺了它，用户会发现自己每次打开行程页都要重新移除同一篇日记。

### 30.5 行程页的形态

行程页会是全站信息密度最高的一页：

1. **顶部**：地图，画出这趟的行程段与城市（第 13 章的地图组件，只是数据被限定为本行程）。
2. **中部**：按天的时间线，自动聚合那几天的日记、随想、行程段。
3. **底部**：图集与总结正文。

它也顺手解决了日历的问题——**行程是区间**，在日历上是横跨多天的色条而非一个点。

### 30.6 轻量验证选项

若想先验证需求再投入实现，可以**先不建实体**：给所有相关记录打同一个标签（`#2024关西`），标签页就是简易行程页。

零成本，用两趟就知道是否需要完整版。

> 这条写进文档是为了保留选择权。真做起来时，如果标签方案已经够用，就不必为了「文档里写了」而实现完整实体。

## 31. 数据模型关键决策（v1.9 新增）

> 本章的决定**改起来最贵，必须在录入真实数据前定死**。
> 等有了几百条记录再改，就不是改代码，是数据迁移。
> 每一节都建议落成一条 ADR。

### 31.1 国家用 ISO 3166 三位码 **[建议写成 ADR]**

**数据库存三位码（`KOR`、`JPN`），不存国家名字。**

TopoJSON 中的国家按数字码或三位码索引，而「韩国 / 南韩 / Korea, South / Republic of Korea」这类不一致会让匹配变成灾难。录入时归一化成 ISO 码，显示时查表转中文名。

`lib/footprint.ts` 中已有的「同名不同国」歧义处理，正好可用国家码消解。

> **必须在录入真实数据前定下来。**

### 31.2 行程段统一模型 **[建议写成 ADR]**

见第 14 章。要点：

- `行程段 = mode + 一组有序停靠点`，飞机与火车共用一张表
- 加新交通方式只是加一个枚举值
- 唯一实质差别是**陆路贴地面走**，渲染用折线而非大圆弧
- 停靠点粒度：飞行用机场，陆路用城市

### 31.3 行程（Trip）实体 **[建议写成 ADR]**

见第 30 章。要点：

- 行程是容器，不存原始数据
- 与「系列 > 文件夹 > 内容」同构，可复用已有的多对多逻辑
- 日期是**区间**而非单点

### 31.4 通用关联表 **[建议写成 ADR]**

行程要关联日记、随想、行程段、笔记；将来还会有别的组合。**不要为每一对关系建一张表**，而是建一张通用关联表：

| 列 | 说明 |
|---|---|
| `source_type` / `source_id` | 关联的发起方，例如 `trip` |
| `target_type` / `target_id` | 关联的目标，例如 `diary` |
| `relation` | 关系类型，例如 `includes` |
| `origin` | `auto`（系统建议）/ `manual`（人工添加） |
| `excluded_at` | 人工排除的时间；非空表示「不要再自动加回来」 |
| `created_at` | |

**`origin` 和 `excluded_at` 这两列是第 30.4 节那条规则的落点**——没有它们，「解除后不被自动加回来」实现不了。

> 通用关联表的代价是查询要多一次 join，且类型安全靠约定而非外键。
> 对这个规模的项目，换来的「加一种关联不用改表结构」更值。

### 31.5 所有内容类型统一带日期字段

时间线、日历、行程自动聚合全都建立在这个之上。

- 日记：天然有日期。
- 随想：目前只有创建时间，**需要显式的「发生日期」字段**，允许与创建时间不同（补记昨天的事）。
- 笔记与文章：发布日期。
- 行程段：出发日期。
- 轨迹：日轨迹是单日期，段轨迹是区间（第 12.4 节）。
- 行程：区间。

> 统一之后，「把某段时间发生的一切拉出来」才是一次查询，而不是每种类型写一遍。

## 32. 地图技术方案（v1.9 新增）

### 32.1 不使用任何地图 API

**不需要 Google Maps 或任何地图服务。**

要画的是「去过哪些国家、哪些城市、飞过哪些航线」，这类**世界尺度示意图**只需国家轮廓矢量数据，不需要街道、POI、楼层级瓦片。用瓦片地图反而让画面又乱又丑。

**为什么不用 Google Maps**：要 API key、要绑信用卡开计费、key 暴露在前端、开源仓库管理密钥麻烦、国内访问不稳定。对本场景全是成本没有收益。

### 32.2 选定方案：Natural Earth + d3-geo，纯前端 SVG 渲染 **[建议写成 ADR]**

- 数据用 `world-atlas` 的 TopoJSON，110m 精度约 100KB，直接打包进仓库
- `d3-geo` 做投影，`d3.geoInterpolate` 画大圆航线
- **零 API key、零费用、零请求、零限流、离线可用**，别人 fork 下来直接能跑

**顺带解决了「找不到好看的地图」的问题**：矢量渲染意味着颜色自定。把设计 token 里那套灰绿色套到海洋与陆地上，去过的国家用主色高亮，效果会比任何现成地图更像「自己的作品」——现成瓦片地图永远是别人的配色。

**库的选择**：直接用 `d3-geo` + `topojson-client`，不要引 `react-simple-maps`。项目当前没有任何 UI 库，且这个交互需要细粒度控制，自己写反而更省事。

### 32.3 内置数据源（同样不需要接口）

| 用途 | 数据集 | 许可 |
|---|---|---|
| 国家轮廓 | Natural Earth / world-atlas TopoJSON | 公共领域 |
| 机场坐标 | OurAirports 或 OpenFlights `airports.dat`（含 IATA/ICAO 码与经纬度） | 公共领域 |
| 城市坐标 | GeoNames `cities15000`（两万余个城市） | CC BY，**需署名** |

用户输 `PVG`，本地查表出坐标，航线自动画出。

> 在「About」页写一行数据来源署名。GeoNames 的 CC BY 要求如此，而且这也是专业度的体现。

### 32.4 交互形态：自由缩放平移为主，点击国家为快捷方式

**两种方案不冲突，应叠加**：底层是可自由缩放平移的地图，点击某国时**仍然**平滑飞过去并 fit 到该国。这就是所有地图应用的实际做法——点击只是自由缩放之上的一个快捷方式，不是另一套逻辑。

**自由缩放的重大好处**：不需要「必须缩放到某国包围盒」，因此 32.7 节那一堆边界情况**大部分直接消失**。包围盒覆盖表仍然需要，但**不再是必需品**——就算某个国家飞歪了，用户拖两下就能修正，不会卡死。

> 这个差别很关键：从「必须处理对」降级成「处理不好也不致命」。

### 32.5 投影选择：必须做的取舍 **[建议写成 ADR]**

| 投影 | 优点 | 代价 |
|---|---|---|
| 自然地球投影 | 世界视图好看 | 放大到国家尺度后形状明显歪、边缘是弧形，拖动别扭 |
| **Web 墨卡托** | **保角**，任何缩放级别下局部形状都正确 | 世界视图下格陵兰比非洲还大 |

**想做 Google Maps 那种手感，就必须接受墨卡托**，这是必付的代价。

### 32.6 实现要点

**缩放与渲染性能**

- transform 施加在外层 `<g>` 上，**不要每帧重新投影路径**——前者是 GPU 合成，后者会卡。
- 国界线用 `vector-effect: non-scaling-stroke`，否则描边跟着放大。
- 城市点半径写成 `r / k`（k 为当前缩放倍数），否则放大后变成大圆饼。
- 用 `d3.zoom` 的 `translateExtent` 限制平移范围，别让地球被拖出屏幕。

**按缩放级别切换数据精度**

- `k < 3` 用 110m，再往上换 50m（110m 在国家尺度下海岸线明显是折线）。
- 城市点与标签也设阈值：世界视图只显示国家高亮，放大到一定程度才浮现城市，否则一屏几百个点糊成一片。

**大圆航线**

- `d3.geoInterpolate(起点, 终点)` 采样几十个点拼成 GeoJSON LineString，交给 `geoPath` 渲染。
- **关键：不要自己算 SVG 路径。** 交给 `geoPath` 的话，跨太平洋航线（如 PVG→LAX）会被投影裁剪逻辑自动从 180 度经线切成两段画在地图两侧；自己算就会是一条横穿整张地图的直线——**这是所有人做航线图的第一个坑**。
- 墨卡托下大圆弧会明显往北弓（浦东飞法兰克福会擦着西伯利亚过去）。这是**正确的，而且很好看**。
- **重复航段要聚合**：按次数映射线宽或透明度，飞得多的航线更粗更亮。改动很小，但这是整张图里最有信息量也最好看的部分。
- **缩放时重新采样**：世界视图下 50 个点很平滑，放大 20 倍后就是折线了，超过某缩放级别需提高采样密度。
- **机场点与城市点是两套数据，不要混**：航线连的是机场（PVG / SHA），足迹记的是城市（上海），同时显示会重叠。建议航线图层打开时才显示机场点，且样式与城市点区分。

**陆路与飞行的视觉区分**

飞机是弓起的弧线，陆路是贴地的折线，颜色再分开。两种线型在同一张图上比清一色弧线更有信息量、也更好看。

**「高亮正在看的国家」要小心**

需在每次平移时反投影视口中心点、判断落在哪个国家的多边形内并节流。技术上可行，但体验上容易**闪烁**——地图一动高亮就跳，边境地区尤其乱。

> 更稳的做法：**悬停高亮**（桌面）+ 常驻侧边栏列出去过的国家与城市，点列表项地图飞过去。
> 信息量更大也不抖。手机无 hover，靠侧边栏。

### 32.7 若采用「点击国家 fit 到包围盒」，必然撞上的边界情况

- **跨 180 度经线**：俄罗斯、斐济、美国（阿拉斯加）、新西兰。`bounds()` 会返回横跨整个地球的框。
- **有海外领地**：法国（法属圭亚那、留尼汪）、荷兰（加勒比）、英国、美国（夏威夷、关岛）。包围盒同样横跨半个地球。需维护覆盖表指定只缩放到本土。
- **微型国家**：新加坡、摩纳哥、梵蒂冈。需设最大缩放上限，城邦国家直接跳过「国家→城市」这一层。
- **高纬度国家**：俄罗斯、加拿大、挪威在纯 transform 缩放下会拉得很难看。想每国都端正需用 `projection.fitExtent()` 重算投影，效果好但过渡动画更难做。

### 32.8 架构：只有一个地图组件

**城市点与航线只是它的两个图层**，同一个投影、同一个 zoom transform，航线只是多一个 `<g>`，不需要任何新基础设施。

**路由顺势整理**：`/{username}/map` 与 `/{username}/flights` 挂同一个地图组件，只是默认开启的图层不同，再给用户一组图层开关。代码少一套，体验多一层。

**把缩放状态做成路由**：`/{username}/map` 与 `/{username}/map/JPN`。React Router 7 下成本很低，换来浏览器前进后退可用、链接可分享、刷新不丢状态。

> 地图类交互不做这一步，用户按返回键会直接退出站点。

**纯逻辑抽到 `lib/`**：包围盒计算、跨经线处理、ISO 匹配、城市按国家分组——均不依赖 DOM，可像现有 `lib/footprint.ts` 那样单元测试。

> **跨 180 度经线那几个 case 写成测试尤其值，手动验证一定会漏。**
