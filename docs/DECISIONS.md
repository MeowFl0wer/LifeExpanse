# 架构决策记录

这份文档记录 LifeExpanse 开发过程中做过的关键取舍：**当时面对什么问题、为什么选了现在这条路、放弃了什么、后来是否被推翻**。

只记录「换一条路结果会明显不同」的决定。纯粹的实现细节不写在这里，代码注释里说清楚就够了。

格式：每条包含 **背景 / 决定 / 代价 / 状态**。

---

## ADR-001 权限判断下沉到数据层，而不是放在页面里

**背景**  
最早每个页面自己过滤内容：列表页判断 `visibility === 'public'`，详情页再判断一次。结果是同一条规则散在十几个地方，**只要有一个页面忘了写，私密内容就会露出去**。

**决定**  
所有读取都必须经过 `src/api/pkm.ts`，权限过滤在那里做一次，页面拿到的就已经是「这个访客能看的」。页面不允许直接 import 内容数组。

**代价**  
页面全部改成异步，约 15 个组件测试要包 `waitFor`。写起来比直接读数组啰嗦。

**状态**  
生效。但这条规则**被违反过三次**——按 slug 查详情、文件夹索引页、查询串直接下钻，每次都是漏了一个读取入口。教训是：光有约定不够，得让「绕过去」变成编译错误（见 ADR-002）。

## ADR-002 用编译错误代替口头约定

**背景**  
删除的内容仍然出现在首页和面板上。当时的说法是「内容已经移出 `allContent`，所以自然会消失」——但首页读的根本不是 `allContent`，而是原始的种子数组。同一个原因还导致**新建的内容永远不会出现在首页**。

**决定**  
把种子数组从 `mockData.ts` 的导出中去掉，只保留 `allContent` 这一个出口。

**代价**  
无。

**状态**  
生效，且当场见效：改完编译器直接列出了两个越界的文件——比人工排查可靠。

> 这条是整个项目里最重要的一条方法论：**能靠类型系统保证的，就不要靠记性。**「默认就是对的」比「记得要做对」强。

## ADR-003 「不存在」和「无权限」返回同一个 404

**背景**  
如果私密内容返回 403、不存在返回 404，攻击者可以靠遍历 slug 来判断「哪些内容存在但我看不到」，从而推断出标题结构和发布节奏。

**决定**  
两种情况返回完全一致的 404 和同一句文案。前后端都是这个规则；后端在 `services.visible_to` 一处判定。

**代价**  
调试时不容易区分是写错了 id 还是权限不足。用日志补偿，不通过响应体区分。

**状态**  
生效。评论删除接口也遵循同一规则。

---

## ADR-004 API 层作为「换后端」的接缝

**背景**  
需要写后端，但前端已经有 300 多个测试和十几个页面。如果页面直接 `fetch`，那么接后端的那天所有页面和所有测试都要同时改。

**决定**  
`src/api/pkm.ts` 里每个函数第一行判断 `usingBackend()`：配了 `VITE_API_BASE` 走 HTTP 实现（`pkmHttp.ts`），否则走内存实现。**页面调用的函数名和签名完全不变。**

**代价**  
两份实现要保持行为一致，规则改动得改两处（评论权限规则就是这样，前后端各写了一遍）。内存实现不是权威，只是原型。

**状态**  
生效。事实证明是对的：接后端那次**没有改动任何一个页面文件**，测试也不需要 mock HTTP。

## ADR-005 双向链接在前端解析，不搬到后端

**背景**  
接后端后，链接图仍然从本地内存算，导致后端内容不参与反链。要修就得决定：Markdown 链接解析放哪一边？

**决定**  
解析留在 TypeScript（`lib/wikilink.ts`），但**候选集合改成从服务器要**——`/content` 已经按访客过滤，所以能解析出来的都是访客本来就能看的，看不到的会显示成「未解析」而不是泄露标题。

**为什么不搬到后端**  
那意味着 Python 里再写一个 Markdown 链接解析器。两个解析器一定会慢慢跑偏，而这个解析器是已经有测试覆盖的那个。**权限归服务器，解析归一处**，两件事分开处理。

**代价**  
要把作者的可见内容列表整个拉下来。对个人站的数据量可以接受；如果内容量涨到几千条，这里要改成后端出接口。

**状态**  
生效，带已知规模上限。

---

## ADR-006 草稿本地和服务器双写

**背景**  
自动保存最初只写 `localStorage`。换设备就丢，清缓存也丢。

**决定**  
同时写本地和服务器。读取时优先服务器，服务器不可用则回退本地。保留 14 天。

**为什么不只写服务器**  
断网时用户还在打字，这时候「保存失败」等于让人白写。本地那份是断网兜底，不是缓存。

**代价**  
两边可能不一致，需要按时间戳取新的那份。

**状态**  
生效。草稿的 key 一开始没有按用户区分，导致同一浏览器上换账号会读到别人的草稿——已修，key 改为 `owner:edit:id`。

## ADR-007 系列 > 文件夹 > 笔记，但归属是多对多

**背景**  
一开始文件夹和系列只是筛选标签。但一篇文章往往同时属于多个主题，而系列在语义上又比文件夹更上层。

**决定**  
- 内容可以同时属于多个文件夹和多个系列
- 文件夹也可以属于多个系列
- **如果内容所在的文件夹已经在某个系列里，那个系列在选择器里显示为「随文件夹」并锁定**

**为什么要锁定**  
否则同一篇笔记会既在系列里的文件夹中出现，又散落在系列根下，同一个东西看起来像两个。继承来的归属不允许手动重复勾选。

**代价**  
`normaliseMembership` 这个规则前后端各实现了一次，必须同步。后端有对应测试守着。

**状态**  
生效。

## ADR-008 软删除（回收站）而不是直接删

**背景**  
删除是不可逆操作，而这个项目存的是日记和人生记录——误删的代价远高于多占一点存储。

**决定**  
删除写 `deleted_at`，所有正常查询都走 `live_content_query` 过滤掉。回收站里可以恢复，超过保留期才真正清除。

**代价**  
每个查询都得记得过滤。**这一点用 ADR-002 的方法处理**：不暴露裸查询，只暴露 `live_content_query`。

**状态**  
生效。

## ADR-009 会话用 HttpOnly Cookie + 不透明 token，不用 JWT

**背景**  
需要登录态。JWT 是更常见的选择。

**决定**  
服务端生成不透明随机 session id，存表，放 HttpOnly Cookie。前端**不解析**身份，身份从 `/auth/me` 取。

**为什么不用 JWT**  
JWT 无法即时撤销——用户点「退出所有设备」时，已签发的 token 在过期前依然有效。需求里有设备管理这一项，所以撤销能力是硬要求。存表的 session 删一行就失效了。

**代价**  
每个请求多一次数据库查询。这个量级完全不是问题。

**状态**  
生效。「记住我」只影响 Cookie 的 `max_age`，不影响服务端有效期判断。

## ADR-010 前端和后端在同一个仓库，通过 nginx 同源部署

**背景**  
可以拆成两个仓库，也可以放一起。

**决定**  
单仓库（monorepo），`docker-compose` 一起起，nginx 反代到同源路径 `/api/v1`。

**为什么要同源**  
会话是 Cookie。跨域意味着要处理 `SameSite=None`、CORS 预检，并且 Safari 的 ITP 会直接拦第三方 Cookie。同源部署把这一整类问题消掉了。

**代价**  
前后端无法独立部署。对个人项目不是问题。

**状态**  
生效。

## ADR-011 先建 CI，再继续做功能

**背景**  
到 v0.6.0 为止，测试是手动跑的。**只要有一次忘了跑后端测试，坏代码就进主干了**，而且是事后才发现。

**决定**  
`.github/workflows/ci.yml`，每次 push 和 PR 跑：前端类型检查 + 测试 + 构建，后端测试。用 `--frozen-lockfile` 保证 CI 装的版本和干净 clone 一致。

**代价**  
无。

**状态**  
v0.7.0 起生效。

## ADR-012 服务器端加密的范围，以及为什么内容正文不加密

**背景**
「数据在服务器要加密存储」。但「加密」在不同位置含义不同，范围定错要么白做，
要么把功能弄坏。

**决定**
分三层，各用各的手段：

| 数据 | 手段 | 为什么 |
|---|---|---|
| 密码 | Argon2id 哈希 | 永远不需要读回原文 |
| 验证码、恢复码 | 带密钥的 SHA-256 哈希 | 同上，且高熵、校验频繁，用 Argon2 只是徒增延迟 |
| TOTP 密钥 | AES-256-GCM 加密 | **必须能读回原文**才能校验动态码，所以只能加密不能哈希 |
| 内容正文 | **不在应用层加密** | 见下 |

**为什么正文不在应用层加密**

1. **关键词搜索是在 SQL 里对正文做的**。列级加密会直接废掉搜索，
   除非另建一套明文索引——那等于把加密绕开了。
2. **防不住真正的威胁**。密钥在服务器环境变量里，拿到运行中的服务器就能拿到密钥。
   应用层加密防的是**被偷走的数据库文件或备份**，不是被攻陷的服务器。
3. **真正的端到端保密另有其人**：加密空间那个功能就是干这个的，
   密码不上服务器，那才是名副其实的加密。

所以正文在磁盘层面加密（部署时的卷加密），不在列层面加密。

**代价**
- `LIFE_SECRET_KEY` 变成关键材料：换掉它，所有 TOTP 密钥和未使用的验证码全部作废。
  已写进 `.env.example` 的警告里，必须和数据库一起备份。
- 卷加密是部署职责，不是代码能保证的，需要在部署文档里写清楚。

**状态**
生效。

## ADR-013 账号相关接口一律不确认「这个邮箱是否存在」

**背景**
原来的注册接口在邮箱重复时返回 409「该邮箱已被使用」。
任何人都能拿一份邮箱列表刷这个接口，问出「谁在这个站有账号」。

**决定**
所有陌生人能触达的账号接口，存在与不存在返回**完全相同**的响应：

- 请求注册验证码 → 恒定「如果该邮箱可以使用，我们已发送验证码」
- 忘记密码 → 恒定「如果该邮箱存在，我们已发送验证码」
- 注册时邮箱已被占用 → 报**验证码错误**，而不是邮箱冲突
- 限流被触发时，响应也不变——否则限流状态本身成了探测信号

**用户名是例外**：用户名就是公开的主页地址，说它被占用不比访问 `/{username}` 多泄露任何东西。

**代价**
用户填错邮箱时得不到明确提示，只能等不到验证码。这是刻意的取舍。

**状态**
生效，且五条削弱实现的反向验证都被对应测试抓到。

## ADR-014 迁移在应用自己的引擎上跑，不让 Alembic 另开一个

**背景**
从 `create_all` 换到 Alembic。默认写法是 `alembic upgrade head` 自己
`engine_from_config` 建一个引擎。

**决定**
应用启动时迁移，把**应用自己的连接**通过 `config.attributes["connection"]`
交给 Alembic；`env.py` 优先使用这个连接，没有才自己建（供命令行独立使用）。

**为什么**
测试用的是内存 SQLite。Alembic 自建引擎意味着另一条连接——
而对内存 SQLite 来说，**另一条连接就是另一个数据库**。
迁移建的表落在一个转瞬即逝的库里，应用自己的库里一张表都没有。
131 个测试当场全红。

共用引擎让这类错配在结构上不可能发生，不是靠记得配对。

**代价**
`env.py` 要处理两条路径（应用内 / 命令行）。

**状态**
生效。`alembic upgrade head`、`downgrade base` 和应用启动三条路径都验证过。

## ADR-015 端到端验证抓到了两个测试抓不到的 bug

**背景**
不是一条决策，是一条教训，值得单独记下来。

这一轮有两个 bug，**单元测试全绿也照样存在**，都是起真实服务器跑流程时才暴露的：

1. **控制台邮件的验证码从未出现在任何地方**。
   邮件以 `INFO` 记录，但项目里没有任何地方配置日志处理器。
   pytest 自带 handler 所以测试看得见；真实部署下开发者永远收不到验证码。
2. **迁移把整个进程的日志关掉了**。
   Alembic 的 `fileConfig` 默认 `disable_existing_loggers=True`。
   在应用内跑迁移，等于启动时把 uvicorn 和应用的日志全部禁用——
   连上面刚修好的验证码输出也一起没了。

**共同点**：两个都是「测试环境替我们做了某件事」造成的盲区。
pytest 配好了日志，所以测试看不见问题；测试不跑迁移，所以看不见第二个。

**结论**
**涉及进程级配置（日志、启动顺序、迁移）的改动，必须起真实服务器跑一遍。**
测试绿不能替代这一步。

**状态**
已纳入流程：这一轮之后每个 PR 都跑一次端到端脚本。

## ADR-016 上传的文件类型由字节决定，路径由我们决定

**背景**
实现文件上传。客户端会提供两样东西：`Content-Type` 头和文件名。
两样都是客户端说了算。

**决定**

1. **类型看字节**（`storage.sniff`），不看 `Content-Type`，也不看扩展名。
   只认 JPEG / PNG / GIF / WebP / MP4 / WebM 的magic number。
2. **磁盘路径由我们自己的随机 id 生成**，客户端文件名只用于展示。
3. 返回时带 `X-Content-Type-Options: nosniff` 和 `Content-Disposition: inline`。

**为什么**

- 相信 `Content-Type` 意味着任何人都能存一个标着 `image/png` 的 HTML 文件，
  之后再让服务器把它原样吐回来——**这就是存储型 XSS**。
- 用客户端文件名拼路径意味着 `../../../../etc/passwd` 是一个合法上传。
  从自己的随机 id 生成路径，让这一整类攻击在结构上不可能，而不是靠过滤。

**代价**
sniff 只认有限的几种格式，加新格式要改代码。这是刻意的：
白名单比黑名单安全。

**状态**
生效。反向验证过：改成相信 `Content-Type` 或用文件名拼路径，都会有测试失败。

## ADR-017 上传的字节存磁盘，不存数据库；默认私密

**背景**
文件存哪里，以及谁能读。

**决定**
- 字节存磁盘（`LIFE_MEDIA_ROOT`，指向挂载卷），数据库只存元数据
- 每个文件有自己的 `visibility`，**默认 private**
- 读取接口检查 visibility；不存在、已删除、无权限**返回同一个 404**（ADR-003）
- id 是 24 字节随机串。这不是访问控制，是**下限**——不可猜的 URL 之上还有权限检查

**为什么不存数据库**
数据库备份会变得巨大，而媒体文件本来就适合单独同步。
换成对象存储时只需重写 `storage.py` 一个文件。

**为什么默认私密**
上传这个动作本身不应该让文件对全世界可读。
公开是一个明确的选择（发文章时、设封面时），不是默认。

**头像是例外**：头像必须让访客看见，所以上传时就是 public；
而且不受管理员授予的图片权限限制——头像是「有个账号」的一部分，不是内容特权。

**代价**
私密文件不能走 CDN 直连，必须经过应用。个人站规模下无所谓。

**状态**
生效。删除是数据库软删 + 磁盘硬删：用户要求删除后还留着字节，是保留了错误的那一半。

## ADR-018 文件与内容的关联从正文推导，不由客户端声明

**背景**
上传的文件必须属于某条内容，不能有孤儿文件。

**决定**
每次保存内容时**重新读一遍正文和封面**，算出引用了哪些文件，据此建立关联。
客户端不需要（也无法）单独声明。

**为什么不让客户端声明**
如果「附加文件」是一次独立调用，那么**任何忘了调用的路径都会静默产生孤儿**。
而忘记是一定会发生的：新建、编辑、导入 Markdown、粘贴、撤销……每条路都得记得。
从正文推导则让正确成为默认——这是 ADR-002 的同一条方法论。

**只能关联自己的文件**：正文里引用别人的 id 不会让你获得那个文件的所有权。

**生命周期**

| 事件 | 文件 |
|---|---|
| 上传但未保存进任何内容 | 未关联，24 小时后清理 |
| 保存进内容 | 关联 |
| 从正文里移除 | 标记为孤儿，保留 7 天 |
| 内容进回收站 | 标记为孤儿（**不删**，内容可能被恢复） |
| 内容恢复 | 重新关联 |
| 内容彻底删除 | 文件一并删除 |

**为什么移除后不立刻删**
误删一张图，7 天内把它加回正文就能恢复。立刻删掉换来的只是省下几天的存储。

**代价**
需要一个清理任务。目前在启动时跑一次，管理后台也可手动触发；
没有引入调度器依赖。

**状态**
生效。端到端验证过完整生命周期，最后磁盘上零残留。

## ADR-019 展示用缩略图，原图按需

**背景**
一篇文章可能有十张照片。每张都用原图，等于为了显示 700px 宽的图片下载十个几 MB 的文件。

**决定**
- 上传时用 Pillow 生成 WebP 缩略图（最长边 720px），存在原图旁边
- 渲染内容时 `<img>` 指向 `?variant=thumb`
- 原图只在读者点击「查看原图」或「下载」时才请求
- **本来就小于 720px 的图片不生成缩略图**——省不下什么，反而多一个文件

**控件位置**：图片右下角，悬停才出现，不遮挡画面。
**触屏没有悬停**，所以在 `hover: none` 的设备上常驻——
够不着的控件不是含蓄的设计，是缺失的功能。

**缩略图与原图权限完全一致**：私密图片的缩略图同样是私密的。

**视频**：抽第 1 秒的画面作为封面。不取第 0 帧——很多视频开头是黑的，
那样等于没有封面。不足 1 秒的短片回退到第一帧。

**动图**：缩略图保留动画（动画 WebP）。静止的首帧会让每个动图看起来都是坏的。

**ffmpeg 从哪来**：`imageio-ffmpeg`，它以 pip 包的形式自带静态二进制。
用系统 ffmpeg 的话，某次部署少装了都要等到有人上传视频才会发现——
把它变成一个普通依赖，就和其他依赖一样由 `pip install` 保证。

**代价**
新增 Pillow 和 imageio-ffmpeg 依赖（后者约 30 MB）。
缩略图在上传时同步生成，很大的视频会让那一次请求慢一些。

**状态**
生效。实测 60 KB 的原图缩到 688 字节；1280x720 的视频封面 5 KB，
且确认是真实画面而非黑帧。
